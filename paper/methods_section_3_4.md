### 3.4 Installation and Inference

The bitstreams produced by Section 3.3 fully specify the quantized model in the RHT basis. To use the quantized model for inference or evaluation, we install the dequantized weights into a fresh fp16 model instance via the following procedure.

**Dequantization.** For each quantized target, we walk the trellis bitstream through the HYB codebook to recover W~_q in the RHT basis, then apply the inverse Hadamard transform with the saved sign vectors s_L and s_R to obtain W_q in the original weight basis. This is a deterministic, lossless operation: the same bitstream always produces the same W_q. The decoder is implemented in fp32 for numerical stability and the result is cast to bf16 to match the model's parameter dtype.

**Model installation.** We instantiate a standard `OlmoeForCausalLM` from the HuggingFace transformers config and load the original fp16 checkpoint, then replace each quantized target's weight in-place with its dequantized value via the model adapter described in Appendix A. The adapter handles two distinct expert layouts produced by transformers depending on the loader path: a fused `OlmoeExperts` layout where all 64 experts share a single `gate_up_proj` parameter of shape (64, 2*I, H), and an unfused `ModuleList` layout where each expert is an individual `OlmoeMLP` module. Both layouts are functionally equivalent; we install per-target weights into either layout transparently.

**Inference.** Once weights are installed, the model is a standard fp16 OLMoE that runs through the unmodified transformers inference path. Quantization in this paper is therefore a *quality preservation* contribution rather than a runtime efficiency contribution: the dequantized weights occupy the same fp16 memory footprint as the original model (~13.8 GB), and inference latency is comparable to the fp16 baseline. Native 2-bit inference kernels — which would enable both the on-disk compression to be reflected in GPU memory usage and would provide inference speedups via reduced memory bandwidth — are deferred to future work. We position the present paper as establishing 2-bit quality feasibility for MoE models; runtime efficiency is the natural next step and is currently in development as a follow-up paper.

**Evaluation protocol.** We evaluate the installed quantized model identically to how the fp16 baseline is evaluated: using the standard transformers inference path with no quantization-aware modifications. This ensures that any perplexity or downstream task gap between the quantized and fp16 configurations is attributable purely to quantization error in the weights, not to differences in the inference pipeline.

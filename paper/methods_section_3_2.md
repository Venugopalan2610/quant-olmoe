### 3.2 Routing-Conditioned Per-Expert Hessian Collection

The defining feature of MoE inference is sparse routing: at each token, the router selects k experts (k=8 in OLMoE) from a pool of N (N=64) and dispatches the token's hidden state only to those experts. As a consequence, each expert sees a different subset of the calibration distribution. We exploit this by collecting a separate input Hessian for each expert from only the tokens that the router actually dispatches to it.

**Hook mechanism.** OLMoE's transformers implementation packs all 64 experts in each layer into a single `OlmoeExperts` module whose `forward` method iterates over the experts that received any tokens (the "expert hit set"). For each hit expert, it slices out the routed token subset, applies `gate_up_proj[i]` followed by SiLU-and-multiply, then `down_proj[i]`, and scatters the results back via index-add. We install a per-instance monkey patch that mirrors this computation exactly while accumulating the routed inputs into per-expert Hessian buffers. The patch is bit-identical to the original forward in its outputs (verified by a tripwire that compares hooked vs unhooked layer outputs across the calibration set), so the model's behavior during Hessian collection is indistinguishable from inference.

**Two Hessians per expert.** Each expert's MLP has two distinct input distributions:

- The **gate/up input** is the layer norm output, dimension H = 2048, which feeds the fused `gate_up_proj` matmul. We accumulate `H_e^{gate_up} = X_e^T X_e` where X_e is the (n_routed_tokens, H) matrix of routed inputs, in fp32 for numerical stability. Because `gate_proj` and `up_proj` are halves of a single fused matrix that share inputs exactly, a single Hessian serves both during quantization.
- The **down input** is the SiLU-gated intermediate, dimension I = 1024, which feeds `down_proj`. We separately accumulate `H_e^{down} = Y_e^T Y_e` where Y_e is the (n_routed_tokens, I) matrix of intermediate activations.

This gives 2N = 128 Hessians per layer (N for gate_up and N for down), or 16 × 128 = 2048 distinct expert Hessians for the full OLMoE-1B-7B model.

**Calibration data and accumulation.** We use 2048 sequences of 1024 tokens from the C4 validation set [Raffel et al., 2020] as calibration data, totaling ~2.1M tokens. The Hessian buffers are kept on GPU in fp32 and accumulated incrementally via in-place outer-product additions (`addmm_`); the calibration pass requires a single forward sweep over the full calibration set. After the sweep, we normalize each Hessian by its expert's token count and symmetrize (`H ← (H + H^T) / 2`) to correct for fp32 accumulation drift, then save to disk per (expert, projection) pair.

**Routing imbalance.** Because OLMoE's load balancing loss is not perfect, expert token counts are non-uniform. In our calibration sample, the most-routed expert receives roughly 3.5× the tokens of the least-routed expert. We do not rebalance this; the per-expert Hessian for a sparsely-routed expert is genuinely a less informative summary of its input distribution, but BlockLDLQ's damping (Section 3.3) provides numerical stability and the resulting quantization quality is comparable across experts (Section 5).

**Comparison to per-layer Hessian baseline.** Existing MoE quantization work (MoEQuant, MoPEQ, EAQuant) collects a single Hessian per expert layer by ignoring routing and treating the entire batch's hidden states as the input to a notional "average expert." For our ablation, we approximate this baseline by computing the unweighted mean of our per-expert Hessians, weighted by routing token counts: `H^{layer} = Σ_e n_e · H_e / Σ_e n_e`. This is a generous baseline because it uses exactly the same calibration tokens as our main approach; the only difference is that the per-layer baseline collapses 64 distinct distributional summaries into one. Section 5.1 compares the two configurations.
